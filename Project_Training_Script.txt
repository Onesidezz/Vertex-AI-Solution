PROJECT TRAINING SCRIPT — VertexAI Document Processing API
============================================================

1. WHAT IS THIS PROJECT?
   An enterprise search system that connects Content Manager (TRIM) with Google Vertex AI to enable natural language search across company documents.

2. SOLUTION STRUCTURE (4 Projects)
   Four .NET 8 projects: Core, Infrastructure, API, and Tests — each with a specific responsibility.

 Tier 1 — Presentation (API Layer)

  DocumentProcessingAPI.API
  - Controllers, Middleware, Program.cs
  - Only handles HTTP requests/responses, no business logic

  ---
  Tier 2 — Business Logic (Infrastructure Layer)

  DocumentProcessingAPI.Infrastructure
  - All services: DocumentProcessor, GeminiEmbeddingService, RecordSearchService, etc.
  - All the actual work happens here

  ---
  Tier 3 — Data (PostgreSQL + TRIM)

  - PostgreSQL via EF Core + pgvector
  - Content Manager via TRIM SDK


3. DocumentProcessingAPI.Core
   Contains all interfaces, entities, DTOs, and configuration — no implementation, no external dependencies.

4. DocumentProcessingAPI.Infrastructure
   Contains all service implementations, database access, TRIM SDK usage, and Google AI integration.

5. DocumentProcessingAPI.API
   The ASP.NET Web API layer — controllers, middleware, Program.cs, and appsettings.json.

6. THE DATABASE — PostgreSQL + pgvector
   A PostgreSQL database (DocEmbeddingsVertex) that stores vector embeddings and metadata using the pgvector extension.

7. THE EMBEDDINGS TABLE
   The main table — each row is one chunk of one CM record, storing a 3072-dimension vector plus all metadata.

8. WHAT IS AN EMBEDDING?
   A float array of 3072 numbers that represents the semantic meaning of a piece of text, generated by Gemini.

9. WHAT IS CHUNKING?
   Breaking a large document into smaller overlapping pieces (1000 tokens each, 200 token overlap) before embedding.

10. TEXT EXTRACTION — DocumentProcessor
    Extracts plain text from PDF, DOCX, DOC, XLSX, PPTX, TXT, CSV files using iText7 and OpenXML libraries.

11. CHUNKING — TextChunkingService
    Splits extracted text into overlapping chunks at sentence boundaries to preserve context.

12. EMBEDDING GENERATION — GeminiEmbeddingService
    Calls Google Vertex AI (gemini-embedding-001) via gRPC to convert each text chunk into a float[3072] vector.

13. BATCH PROCESSING — RecordEmbeddingService
    Orchestrates the full pipeline: fetch records from CM → extract text → chunk → embed → save to PostgreSQL.

14. SMART SYNC — SyncCheckpoint
    Tracks the last sync date and page so the job can resume after failure and skip unchanged records.

15. HYBRID SEARCH — RecordSearchService
    Combines keyword search (TRIM IDOL index) and semantic search (pgvector cosine similarity) to find best results.

16. QUERY ANALYSIS — RecordSearchHelperServices
    Extracts dates, file types, keywords, and sort intent from a natural language query before searching.

17. GOOGLE AI HELPERS — RecordSearchGoogleServices
    Calls Gemini to extract keywords from queries and synthesize human-readable answers from search results.

18. AI SUMMARY & Q&A — RecordAIController
    Collects all chunks for a record and asks Gemini to summarize or answer a specific question about it.

19. CONTENT MANAGER INTEGRATION — ContentManagerServices
    Uses the TRIM SDK to connect to Content Manager, fetch paginated records, and execute IDOL index searches.

20. WINDOWS AUTHENTICATION — WindowsAuthenticationService
    Reads the current Windows user identity and queries Active Directory for display name, email, and group membership.

21. API CONTROLLERS
    Four controllers expose REST endpoints: embedding, search, AI summary/Q&A, and Content Manager operations.

22. RATE LIMITING
    Limits API calls per IP (100/min general, 50/min search, 10/min upload) using AspNetCoreRateLimit.

23. SCHEDULED SYNC — Quartz Job
    A background job runs on a cron schedule (default: every hour) to automatically sync and embed new/changed records.

24. LOGGING — Serilog
    Structured logging to both console and daily rolling log files under the /logs folder.

25. CONFIGURATION — appsettings.json
    Central config for Vertex AI credentials, TRIM connection, PostgreSQL connection, sync schedule, and rate limits.

26. END-TO-END FLOW (INDEXING)
    CM Record → Extract Text → Chunk → Generate Embedding → Save to PostgreSQL (one row per chunk).

27. END-TO-END FLOW (SEARCH)
    User Query → Analyze → Keyword Search (TRIM) + Vector Search (pgvector) → Merge Results → Gemini Answer.

28. STORAGE SIZE RULE OF THUMB
    100 GB of raw documents produces approximately 130–160 GB in PostgreSQL (vector dominates at ~12 KB per chunk).

29. ProcessAllRecordsAsync — Batch Sync Pipeline

    Scheduler (Quartz.NET):
    Triggers the sync job on a cron schedule (default: every hour). Can be paused, resumed, or manually triggered
    via REST API (/api/RecordSyncScheduler). Currently disabled — Quartz registration is commented out in Program.cs.

    GetOrCreateCheckpointAsync:
    Loads the last saved sync state from PostgreSQL before each run. Stores LastSyncDate and LastProcessedPage
    so the job resumes from where it left off after a crash instead of restarting from scratch.

    GetRecordsPaginatedAsync:
    Fetches records from Content Manager in pages of 1000. If LastSyncDate exists, adds a TRIM date filter
    (updated>=date) to return only records changed since the last run — making each run incremental, not full.

    existingTimestamps (Smart Change Detection):
    Loads stored DateModified timestamps from PostgreSQL and compares with Content Manager's DateModified per record.
    Records with matching timestamps are skipped — only truly changed records are re-embedded, saving API calls.

    SemaphoreSlim (Parallel Processing):
    Processes up to 10 records simultaneously per page using a ticket system (SemaphoreSlim).
    All tasks are queued instantly but only 10 run at a time. Waits for the full page to finish before moving on.

30. SearchRecordsAsync — Semantic Search Pipeline

    CalculateDynamicSearchLimit:
    Fetches more results from pgvector than the user asked for (e.g. topK=10 → fetch 200) to account for
    post-filtering by date/file type. Multiplier grows with query complexity (earliest/latest=20×, date filter=10×).

    CalculateDynamicMinimumScore:
    Lowers the similarity score threshold for broad query types — technical terms, proper nouns, document type words.
    Prevents valid results from being excluded due to a threshold that is too strict for general queries.
